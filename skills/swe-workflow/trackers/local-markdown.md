# Tracker: Local Markdown Files

**CLI**: none — issues are markdown files on disk.
**Convention**: mattpocock's `.scratch/<feature-slug>/` layout (see [setup-matt-pocock-skills/issue-tracker-local.md](https://github.com/mattpocock/skills/blob/main/skills/engineering/setup-matt-pocock-skills/issue-tracker-local.md) — authoritative).
**ID format**: full path to the issue file, OR `<feature-slug>/<NN>` when context is unambiguous.

## Repo layout

Issues and PRDs live under `.scratch/`, **grouped by feature** — this preserves the `/to-prd` → `/to-issues` parent/child relationship in the filesystem:

```
.scratch/
├── <feature-slug>/
│   ├── PRD.md
│   └── issues/
│       ├── 01-<slug>.md
│       ├── 02-<slug>.md
│       └── ...
└── <another-feature-slug>/
    ├── PRD.md
    └── issues/
        └── ...
```

- One **feature directory** per PRD.
- Implementation issues numbered from `01` within each feature's `issues/` subdir.
- `/to-prd` creates `PRD.md`; `/to-issues` creates the numbered files; `/triage` updates the `Status:` line.

## File format

```markdown
# Truncate descriptions at word boundary

Status: ready-for-agent

When descriptions exceed 1024 chars, they truncate mid-word, producing
broken output like "Use when the user wants to confi".

## Acceptance criteria

- [ ] Descriptions under 1024 chars are unchanged
- [ ] Over 1024 chars truncated at last word boundary
- [ ] Truncated descriptions end with "..."

## Comments

> *This was generated by AI during triage.*
> ...
```

- The first H1 is the title.
- A `Status:` line near the top carries the triage role.
- The **body itself IS the brief** (no separate `## Agent Brief` section — the whole file is structured as one).
- Comments and conversation append at the bottom under `## Comments`.

## Triage role strings

The `Status:` line uses canonical mattpocock vocabulary (see [setup-matt-pocock-skills/triage-labels.md](https://github.com/mattpocock/skills/blob/main/skills/engineering/setup-matt-pocock-skills/triage-labels.md)):

- `needs-triage` — maintainer needs to evaluate
- `needs-info` — waiting on reporter for more info
- `ready-for-agent` — fully specified, ready for an AFK agent
- `ready-for-human` — needs human implementation
- `wontfix` — will not be actioned

Don't re-document these here — defer to `triage-labels.md` as the source of truth.

## Extracting fields

| Normalized field | How to extract |
|---|---|
| title | First H1 (`# …`) in the file |
| body | Everything between the H1 and `## Comments` (exclusive), minus the `Status:` line |
| labels | The value on the `Status:` line (single label by convention; the role string) |
| agent brief | The body itself IS the brief (the whole file is one well-structured issue). Return as `[{"body": "<entire body>"}]`. |

If a file carries additional label-like lines (e.g. `Type:`, `Priority:`), extract them too — but `Status:` is the only one mattpocock's chain reads.

## ID handling

The user typically passes either:

- A **full path** to the issue file: `.scratch/auth/issues/01-add-login.md`
- The **`<feature>/<NN>`** form: `auth/01` (when unambiguous)

The bootstrap should accept both forms; resolve to a real path before reading.

Branch and worktree naming:

- Branch: `issue-<feature-slug>-<NN>-<slug>` (e.g., `issue-auth-01-add-login`)
- Worktree: `../<repo>-issue-<feature-slug>-<NN>/`

## Auto-detect signal

A `.scratch/` directory exists at the repo root.
